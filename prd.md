## 1. Summary

**GrepLoop** is a self-hosted, multi-tenant code review platform that combines:

1. **Full-codebase indexing** (tree-sitter → call graph → LLM summaries → vector embeddings) — the architectural decision that separates Greptile-tier review quality (82% bug catch rate) from diff-only reviewers (44–54%).
2. **An agentic review loop** — instead of a single LLM call with pre-assembled context, an agent with tools (search the code graph, get callers, read git history) investigates the diff and produces findings backed by evidence chains.
3. **Multi-user collaboration inside a customer's organization** — engineers log into their company's GrepLoop instance, see the repos they have access to, and share review history.

The platform runs entirely inside the customer's own infrastructure (their VPC, their Docker host, their on-prem box). Source code never leaves their network. No GitHub/GitLab tokens, no third-party SaaS dependency, no external LLM call unless the customer explicitly configures one.

**What changed from earlier drafts:** PRD-1 specced a single-user, local-first daemon. PRD-2 layered the Greptile-style indexing and agentic loop on top. This document supersedes both and pivots the *deployment model* from "local daemon" to "self-hosted SaaS" — the review engine and indexing architecture from PRD-1 and PRD-2 carry forward unchanged; the deployment, storage, and user model are new.

## 2. Problem Statement

Tools like Greptile and CodeRabbit provide automated, context-aware code review — but they require pushing code to a remote host and granting a third-party service access to the repository. For client work under NDA, regulated industries (finance, healthcare, defense), air-gapped environments, proprietary codebases, or simply engineering orgs that don't want their diffs touching a SaaS review tool, there's no equivalent.

GrepLoop closes that gap: the same category of full-codebase-aware review Greptile provides, but running inside the customer's own infrastructure. Multiple engineers inside one company can collaborate on reviews, but no byte of source code ever leaves the customer's network.

The wedge against Greptile is structural, not feature-based: Greptile is a public SaaS and cannot easily become self-hosted. GrepLoop is self-hosted first. That's the defensible position.

## 3. Goals & Non-Goals

### Goals (MVP)
- Run as a single deployable unit (Docker image + Postgres) inside customer infrastructure.
- Index entire codebases (tree-sitter → call graph → summaries → embeddings) so reviews have full-codebase context, not just diff context.
- Run agentic review with tool access to the codebase graph, producing findings backed by evidence chains.
- Support multiple users per customer organization, with per-repo access control (some users see Project A, others see Project B, some see both).
- Watch one or more local git repositories (filesystem paths visible from the GrepLoop server) for new branches and trigger review automatically or on demand.
- Support both cloud and local LLM backends, configurable globally, per-org, or per-project.
- Produce structured review reports viewable in a web UI, with full history per branch.

### Non-Goals (MVP — explicitly deferred)
- Posting comments back to GitHub/GitLab/Bitbucket PRs.
- IDE-embedded inline annotations (VS Code extension) — Phase 2 candidate.
- Review of non-text/binary diffs.
- Cross-organization repository sharing (a repo belongs to exactly one org for MVP).
- Public SaaS hosted by us (we ship self-hosted only; if we ever host it ourselves, that's a separate decision with its own considerations).

### Explicitly "shaped for later, not built now"
The MVP is single-organization per installation, but the schema supports multiple orgs from day one to avoid a painful retrofit later. See Section 5.

## 4. Users & Use Cases

**Primary users (MVP):** Engineering teams inside a single company running GrepLoop on their own infrastructure.

**Core use cases:**
1. *"We want every PR to get a context-aware review before merge, without our code leaving our network."* → Auto-inspect mode.
2. *"I want to review only specific branches on demand, not every commit."* → Mention-triggered mode.
3. *"I want to add a new repo for watching in under a minute."*
4. *"I want to see review history for a branch over time."*
5. *"I want my team to share review visibility — some engineers on Project A, others on Project B, leads on both."* → Multi-user with per-repo access.
6. *"I'm a consultant/contractor working across two client companies. I need to log into either client's GrepLoop instance with the same email."* → One user account, multiple org memberships (Section 5).

## 5. Users, Organizations & Permissions

This is the new core of the platform relative to earlier PRD drafts. The review engine is unchanged; the *who can see what* layer is what's added.

### 5.1 The model

```
User                       (provided by Better Auth — email, sessions, profile)
  │
  ├── OrganizationMember   (user_id, org_id, role)   ← many-to-many
  │       │                                              a user can be in multiple orgs
  │       ▼
  │   Organization          (the customer company)
  │       │   owns
  │       ▼
  │   Repository            (a watched project; what PRD-1 called a "watched project")
  │       │   has
  │       ▼
  │   RepositoryMember      (user_id, repo_id, role)  ← many-to-many
  │                                                       some users see repo A,
  │                                                       some see repo B,
  │                                                       some see both
  │
  └── (a single user can appear in many OrganizationMember rows
       and many RepositoryMember rows)
```

### 5.2 Tables (added to the existing Prisma schema)

| Table | Purpose | Key fields |
|---|---|---|
| `User` | A person. Provided by Better Auth. | `id`, `email`, `name` |
| `Organization` | A customer company. | `id`, `name`, `slug`, `createdAt` |
| `OrganizationMember` | Membership join table. **A user can belong to multiple orgs.** | `userId`, `orgId`, `role` (owner/admin/member) — composite PK |
| `Repository.ownerOrgId` | New column on existing `Repository` table. Every repo belongs to exactly one org. | `ownerOrgId` |
| `RepositoryMember` | Per-repo access. **Org membership does NOT automatically grant repo access** — explicit grant required. | `userId`, `repoId`, `role` (reviewer/viewer) — composite PK |

### 5.3 Role definitions

**Organization level:**
- `owner` — billing control, can delete the org, can transfer ownership.
- `admin` — can invite/remove members, can see all repos in the org regardless of `RepositoryMember`, can manage org-level LLM config.
- `member` — can see only the repos they've been explicitly granted via `RepositoryMember`.

**Repository level:**
- `reviewer` — can trigger reviews, view reports, comment on findings.
- `viewer` — read-only access to review reports.

### 5.4 Design decisions (rationale)

1. **Explicit repo access required, not inherited from org membership.** Real engineering orgs have repos that not every engineer should see — client work, security-sensitive code, infrastructure with secrets. The default is closed; admins grant access per repo. (An "open to org" toggle can be added later for non-sensitive repos to reduce admin overhead.)
2. **One user account can belong to multiple orgs.** Consultants, contractors, and people changing jobs need the same login to exist in two companies. This is a one-line schema decision (composite PK on `(userId, orgId)`, no unique constraint on `userId` alone) that is painful to retrofit if you assume one-user-one-org. Slack, Linear, Notion, GitHub all allow this. So does GrepLoop.
3. **One repo, one org.** Cross-org repo sharing is out of scope for MVP. If two companies need to share a repo, they can run separate GrepLoop instances — that's the privacy guarantee they're paying for anyway.

### 5.5 Auth implementation

**Better Auth** is already in the project's dependencies. Its **`organization` plugin** handles orgs, members, roles, invites, and acceptance flows out of the box. Use it rather than hand-rolling. The schema above falls out of enabling the plugin.

The `organization` plugin also includes an **invite flow** — emails sent to invitees with an acceptance link. Rate limits on invite-email resends are built in (default is roughly 3–5 per hour per email). For dev, this rate limit can be raised or disabled in the Better Auth config; for prod, leave it and surface a clear message when the user hits it. The invite state lives in Better Auth's `organization_invitation` table — that's where you'd clear a stuck invite.

## 6. Core Concepts & Terminology

| Term | Meaning |
|---|---|
| **Organization** | A customer company. Owns repositories. Has members. |
| **Repository** | A local git repository registered with GrepLoop for review. Equivalent to what PRD-1 called a "watched project." Belongs to one organization. |
| **Local PR** | A git branch in a watched repo that differs from its base branch and matches the configured criteria. Not a GitHub/GitLab PR object — there is no remote. |
| **Base branch** | The branch a local PR is diffed against (typically `main`), configurable per repo. |
| **Review pass** | One execution of the agentic review loop against a specific diff snapshot. |
| **Trigger mode** | Per-repo setting: `auto` or `mention`. |
| **Index** | The tree-sitter-parsed, LLM-summarised, vector-embedded map of an entire repository's code. Built once on registration, maintained incrementally. |
| **Evidence chain** | The list of files/lines/commits the agent consulted to reach a finding. Required on every finding. |

### 6.1 What counts as a "Local PR"?
A branch is review-eligible when:
- It is not the configured base branch.
- It has at least one commit not present on the base branch.
- It optionally matches a configured branch name pattern (e.g. `feature/*`). If no pattern is set, all non-base branches qualify.

GrepLoop reads `.git` state directly via local git commands. No remote is involved.

## 7. Trigger Modes

### 7.1 Auto-inspect mode
- A filesystem/git watcher polls `.git/refs/heads` and `.git/logs/HEAD` for branch changes (or uses OS-level file watching where available).
- When a qualifying branch is detected, GrepLoop waits for a configurable **quiet period** (default: 5 minutes of no new commits) before triggering review.
- If new commits land after a review has already run, GrepLoop queues a fresh pass after the next quiet period. Report history shows all passes, not just the latest.

### 7.2 Mention-triggered mode
Three trigger mechanisms, in order of friction (low to high):
1. **CLI command / API call** — `greploop review <branch>` from the server, or the equivalent "Review now" button in the web UI. Recommended primary path for MVP.
2. **Marker file** — presence of a `.greploop-review` file (empty, or containing flags) in the branch's working tree triggers review on the next watcher poll, then the marker is consumed.
3. **Commit message marker** — a commit message containing `@PRBot review` (configurable string) triggers review of that branch. Closest offline analog to the GitHub-comment-mention pattern.

### 7.3 Per-repo configuration
Each watched repo has its own settings, stored in Postgres (not environment variables, so multiple repos coexist cleanly):

```yaml
# Stored as a JSON column on the Repository row, or in a related settings table
base_branch: main
trigger_mode: auto              # auto | mention
quiet_period_minutes: 5
branch_pattern: "feature/*"     # optional, omit for "all branches"
llm_backend: inherit            # local | cloud | inherit (use org default)
review_categories:              # subset of Section 14 categories; omit for "all"
  - correctness
  - security
```

## 8. Codebase Indexing Pipeline (Phase 0)

The index is built when a repo is first registered with GrepLoop and maintained incrementally as files change. It runs independently of the review pipeline — a background service that keeps a live map of the codebase so that when a review is triggered, the map is already there.

### 8.1 Stage 1 — Tree-sitter parsing

Parses every source file in the repo into an AST and extracts:
- All function and method definitions: name, parameters (names + types if present), return type, start/end line, docstring if present in source.
- All class and interface definitions: name, parent classes, start/end line.
- All import/require statements: what is imported, from where.
- All function call sites: which function is called, on which line, with what argument names.
- All exported symbols (for module boundary analysis).

**Languages supported at MVP:** JavaScript, TypeScript, Python, PHP, Ruby, Go. Additional grammars installed on demand based on detected file extensions.

**Limitations:** Tree-sitter is a parser, not a type-checker. It doesn't resolve types or dynamic call targets (e.g. `obj[methodName]()`). GrepLoop flags call sites it cannot statically resolve rather than silently dropping them.

### 8.2 Stage 2 — Call graph construction

From parsed symbols, GrepLoop builds a directed call graph:

- **Nodes:** Functions, methods, classes, files, modules.
- **Edges:**
  - `CALLS` — function A calls function B
  - `IMPORTS` — file A imports symbol B from file C
  - `DEFINES` — file A defines function B
  - `EXTENDS` — class A extends class B
  - `OVERRIDES` — method A overrides method B (where detectable)

Edges store their source location (file + line) so the agent can cite them precisely in evidence chains.

**Import resolution:** Follows the language's own module system (e.g. `tsconfig.json` paths for TS, `sys.path` for Python). Unresolvable imports (e.g. external npm packages) are stored as unresolved edges — still useful context, just without a target node.

### 8.3 Stage 3 — Docstring generation (LLM pass)

For each function and class in the call graph, GrepLoop generates a natural-language summary if one doesn't already exist in source.

**Why not embed raw source?** Vector search over raw source is syntactically biased — it finds code that *looks* similar, not code that *does* similar things. Embedding natural-language descriptions produces far more semantically accurate retrieval. This is the same approach Greptile uses and is the reason their retrieval quality is high.

**Prompt per symbol:**
```
Given this function, write a single concise paragraph (2-4 sentences) in plain English
describing what it does, what it accepts as input, what it returns, and any important
side effects or error conditions. Do not describe implementation details unless they
are the only way to convey the function's behaviour.

Function name: {name}
File: {file_path}
Signature: {signature}
Source:
{source_code}
```

**When to run:** First index of a file generates summaries for all its symbols. Incremental updates re-run only for symbols whose source has changed.

**Cost:** This is the most API-call-intensive part of GrepLoop on cloud backends. A codebase with 5,000 functions makes ~5,000 small LLM calls on first index. Batched, rate-limited, runs in the background, amortised over all subsequent reviews. For local backends (Ollama), slow but free.

### 8.4 Stage 4 — Embedding

Each generated summary is embedded using the configured embedding model:

- **Cloud backend:** OpenAI `text-embedding-3-small` or equivalent.
- **Local backend:** `nomic-embed-text` via Ollama, or any GGUF-compatible embedding model.

Vectors are stored alongside symbol metadata using Postgres + **pgvector** (see Section 9).

## 9. Index Store

All index state lives in the same Postgres database as users, orgs, repos, and review history. **One database, no split-brain.** The existing Prisma schema already has `Symbol`, `Edge`, and `File` models targeting this store.

### 9.1 Storage components

**`symbols` table** — every function, class, file, module:

| Column | Type | Notes |
|---|---|---|
| `id` | TEXT PK | Stable hash of `repoId + filePath + name + kind` |
| `repoId` | TEXT FK | References `repositories.id` |
| `filePath` | TEXT | |
| `name` | TEXT | |
| `kind` | TEXT | `function` / `class` / `method` / `module` / `file` |
| `language` | TEXT | |
| `lineStart` | INT | |
| `lineEnd` | INT | |
| `signature` | TEXT NULL | Raw signature string |
| `sourceHash` | TEXT | SHA256 of source — used to detect changes |
| `summary` | TEXT NULL | Generated natural-language docstring |
| `summaryAt` | BIGINT NULL | Unix timestamp of summary generation |
| `embedding` | VECTOR(1536) NULL | pgvector column for semantic search |

**`edges` table** — directed edges between symbols:

| Column | Type | Notes |
|---|---|---|
| `id` | TEXT PK | |
| `repoId` | TEXT FK | |
| `fromId` | TEXT | References `symbols.id` |
| `toId` | TEXT NULL | Nullable for unresolved imports |
| `toRaw` | TEXT NULL | Raw unresolved target string when `toId` is null |
| `kind` | TEXT | `calls` / `imports` / `defines` / `extends` / `overrides` |
| `filePath` | TEXT | |
| `line` | INT | |

**`files` table** — file-level metadata for incremental tracking:

| Column | Type | Notes |
|---|---|---|
| `repoId` | TEXT FK | Composite PK with `filePath` |
| `filePath` | TEXT | |
| `fileHash` | TEXT | SHA256 of file content |
| `parsedAt` | BIGINT | Unix timestamp |

### 9.2 Why Postgres + pgvector?

- **One datastore for everything** — users, orgs, repos, reviews, and the index all live in the same Postgres instance. No separate SQLite/Vector DB to manage, back up, or sync.
- **Supabase already provides this** — pgvector ships with every Supabase Postgres instance. No extension installation needed for customers who pick Supabase.
- **Transactional consistency** — when a repo is deleted, its symbols/edges/files delete in the same transaction (the cascade is already in the schema). With separate stores you'd risk orphaned index entries.
- **Scale** — Postgres + pgvector handles codebases up to millions of symbols without issue at GrepLoop's query patterns. Beyond that, a dedicated vector DB (Qdrant, Weaviate) can be slotted in as a swap.

### 9.3 Incremental update strategy

1. Filesystem watcher (same one used for branch watching) also watches for file changes in registered repos.
2. On any file change, compute `SHA256(file_content)` and compare to `files.fileHash`.
3. If unchanged: skip. If changed or new:
   - Re-parse with tree-sitter.
   - Diff extracted symbols against existing symbols for that file.
   - For changed/new symbols: regenerate summary, re-embed, update graph edges.
   - For removed symbols: delete from `symbols`, `edges`.
   - Update `files.fileHash` and `files.parsedAt`.
4. File deletions prune all associated symbols and edges.

The first index of a large repo is the slow step (minutes to hours depending on size and LLM backend); subsequent updates are incremental and finish in seconds.

### 9.4 Index size expectations

| Repo size | Functions/classes | DB size (symbols + vectors) | First index time (cloud LLM) | First index time (local LLM) |
|---|---|---|---|---|
| Small (<5k LOC) | ~200 | <5 MB | 2–5 min | 10–20 min |
| Medium (5k–50k LOC) | ~2,000 | ~30 MB | 15–30 min | 1–3 hrs |
| Large (50k–200k LOC) | ~8,000 | ~110 MB | 1–2 hrs | 4–12 hrs |
| Very large (200k+ LOC) | ~30,000+ | ~400 MB | 3–6 hrs | 24+ hrs |

For very large repos on local LLM backends, GrepLoop shows progress, allows reviews on partial indexes (with a warning), and never blocks the user from doing other things while indexing runs.

## 10. Retrieval at Review Time

When a diff arrives, GrepLoop pulls the right slice of the codebase into the agent's context. Two complementary strategies run in parallel.

### 10.1 Seed extraction from the diff

```
diff → changed file paths + changed line ranges
     → query symbols table for symbols whose (filePath, lineStart, lineEnd)
       overlaps changed line ranges
     → seed symbol set
```

### 10.2 Graph traversal (structural context)

From each seed symbol, traverse the call graph:
- **Callers (depth 2):** Every function that calls a seed symbol, plus every function that calls those callers. These are most likely to break if the seed's behaviour changes.
- **Callees (depth 1):** Every function the seed calls. Context on what the seed depends on.
- **Co-importers:** Files that import the same modules as the changed files.
- **Class hierarchy:** Full inheritance chain (parents + children) and overrides.

Depth limits are configurable (default: callers depth 2, callees depth 1) because deep traversal of a highly-connected codebase could pull in half the repo. The agent can request deeper traversal as a tool call if it determines the default is insufficient for a specific finding.

### 10.3 Vector search (semantic context)

For each seed symbol's summary, run vector similarity search across all embedded summaries to find functions elsewhere in the codebase that do semantically similar things. This catches the "duplicate utility" class of bug — where a changed function reimplements something that already exists with better edge case handling — which graph traversal alone won't find.

Returns top-K results (default K=10), filtered to exclude symbols already retrieved by graph traversal.

### 10.4 Git history retrieval

For each changed file:
```bash
git log --oneline -20 <file_path>         # last 20 commits touching this file
git log -p -3 <file_path>                 # full diff of last 3 commits
```

Gives the agent: whether this code was recently changed (churn = regression risk), what the previous implementation looked like, and commit messages describing intent.

### 10.5 Context assembly and token budgeting

Retrieved symbols, git history, and diff are assembled into a context payload with a token budget:

| Backend | Usable context budget (leaving room for output) |
|---|---|
| Claude Sonnet (cloud) | ~180,000 tokens |
| GPT-4o (cloud) | ~120,000 tokens |
| Local 7B model (Ollama) | ~6,000 tokens |
| Local 13B model (Ollama) | ~12,000 tokens |
| Local 32B+ model (Ollama) | ~30,000+ tokens |

**Priority order when truncating:** diff always fits first → callers/callees next → semantic results → git history last. For local backends, traversal depth and K are reduced automatically to fit. Every report explicitly notes which backend was used, how much context was available, and what percentage of retrieved context fit — local-model reviews will be genuinely less thorough than cloud reviews for large diffs, and GrepLoop surfaces that honestly rather than hiding it.

## 11. Agentic Review Loop

This is the core differentiator from diff-only reviewers. Instead of one LLM call with pre-assembled context, the review is an agent with tools running a loop.

### 11.1 Why a loop matters

A single LLM call with pre-assembled context is fundamentally limited: you retrieve what you think is relevant before the LLM has seen the diff. The agent approach inverts this — the LLM sees the diff first, forms hypotheses, then *requests* the specific additional context it needs to confirm or refute them. This is closer to how a skilled human reviewer works.

In practice, the agent says: *"This function calls `processPayment` — show me its implementation"* or *"has `auth.ts` been modified recently? Show me the last three commits"* or *"find all functions that interact with the `sessions` table"* — and GrepLoop executes those queries against the index, without the LLM needing all of it pre-loaded.

### 11.2 Agent tools

```
search_codebase(query, limit?) → symbol[]
  Semantic search over embedded summaries. Use for: "find all functions that
  handle authentication", "find utilities that format dates".

get_symbol(file_path, symbol_name) → symbol
  Retrieve a specific function or class by name and file path. Returns full
  source, signature, summary, and generated docstring.

get_callers(symbol_id, depth?) → symbol[]
  Return all functions that call the given symbol, up to depth levels
  (default 2). Use when a changed function's callers need checking.

get_callees(symbol_id) → symbol[]
  Return all functions the given symbol calls. Use when dependencies of a
  changed function need checking.

get_file_history(file_path, num_commits?) → commit[]
  Return recent git log for a file, including commit messages and diffs.
  Default 5 commits.

get_file_contents(file_path, line_start?, line_end?) → string
  Return raw file contents, optionally sliced to a line range.

find_similar(symbol_id, limit?) → symbol[]
  Return semantically similar symbols from across the codebase. Use for
  detecting duplicate implementations.
```

### 11.3 Loop structure

```
INITIAL PROMPT:
  You are reviewing a pull request on a local branch. Your job is to identify
  real bugs, security issues, and correctness problems — not style preferences
  unless they indicate a deeper problem. Focus on issues that could cause
  incorrect behaviour, data loss, security vulnerabilities, or broken contracts
  between components.

  You have tools to explore the codebase. Use them. Do not produce findings
  based on the diff alone if you suspect a cross-file issue — investigate first.
  For every finding you produce, you must be able to cite the specific evidence
  you found. If you cannot cite evidence, do not produce the finding.

  Here is the diff:
  {diff}

  Here is the initially retrieved context (callers, callees, related symbols):
  {initial_context}

LOOP:
  while agent has tool calls pending:
    execute tool calls → return results to agent
    agent continues reasoning
  agent signals DONE → extract findings from final output

MAX ITERATIONS: 10 (configurable — prevents runaway loops on large diffs)
```

### 11.4 Finding format from the agent

```
FINDING:
  category: correctness | security | performance | accessibility | style
  severity: blocker | warning | suggestion
  file: path/to/file.ts
  line: 42
  title: One-line description of the issue
  explanation: Plain English explanation of why this is a problem
  evidence:
    - file: path/to/other-file.ts, line: 88 — function `processPayment` called
      here will receive null when input is undefined, but has no null guard
    - file: path/to/utils.ts, lines: 12-28 — existing `validateInput` utility
      handles this case; consider using it instead
  suggested_fix: |
    Optional short code snippet showing a fix. Mark as suggestion only.
    Do not auto-apply.
  confidence: high | medium | low
```

The `evidence` block is the key addition. Every finding must have at least one evidence entry — a specific file and line reference from the codebase that supports the claim. Findings the agent cannot support with evidence are discarded before report assembly.

### 11.5 Confidence scoring

- **High** — direct code evidence of the bug (e.g. a null dereference path traceable end-to-end through actual code).
- **Medium** — strong circumstantial evidence (a pattern that's usually a bug, in a context where it's likely to manifest).
- **Low** — potential issue but cannot confirm without runtime information.

Low-confidence findings are included but visually separated and explicitly noted as needing human judgment. This controls the false-positive rate without losing real bugs.

## 12. Evidence Chains & Report Format

Each finding in the rendered report includes a collapsible evidence trail:

```markdown
## FINDING: Null dereference in payment handler [BLOCKER] [HIGH CONFIDENCE]

**File:** `src/payments/handler.ts`, line 142
**Category:** Correctness

`processPayment()` calls `getUser(userId)` which can return `null` when the
user session has expired (see auth.ts:88). On line 142, the return value is
passed directly to `chargeCard()` without a null check. `chargeCard()` does
not handle null input (see billing.ts:34) and will throw an uncaught exception,
which the global error handler logs but does not surface to the caller — meaning
the payment appears to succeed to the caller while actually failing.

<details>
<summary>Evidence (3 files traced)</summary>

- `src/auth/session.ts:88` — `getUser()` explicitly returns `null` on session
  expiry: `if (!session) return null`
- `src/billing/charge.ts:34` — `chargeCard()` destructures its input immediately:
  `const { amount, currency } = user` — throws TypeError on null
- `src/payments/handler.ts:142` — call site with no null check:

## 13. Git History Integration

### 13.1 Churn scoring

Files with high recent churn are statistically more likely to contain bugs. Per changed file:

```
churn_score = (commits_touching_file_in_last_30_days) /
              (days_since_file_created + 1)
```

Files above a configurable threshold (default: more than 5 commits in 30 days) get a "high churn" flag, and the agent is explicitly instructed to be more thorough on them.

### 13.2 Regression detection context

For each changed function, GrepLoop retrieves the git diff of that function from the last 3 commits that touched it. This gives the agent the ability to see whether a bug was previously fixed in the same location (and may be being reintroduced), what the previous implementation looked like, and the commit messages that explain *why* the code is the way it is.

This context is pre-loaded (not a tool call) for any changed function with recent history — it's almost always relevant.

## 14. Review Criteria & Metrics

Each category can be toggled on/off per-repo (see `review_categories` in Section 7.3).

### 14.1 Correctness
- Logic errors, off-by-one issues, incorrect conditionals.
- Unhandled error paths / missing error handling.
- Type safety gaps (flagging `any`/untyped escape hatches in typed languages, missing null checks).
- Test coverage gaps for changed logic (flagged, not blocking — GrepLoop reports, it doesn't gate).

### 14.2 Security (OWASP-aligned)
- Injection risks: unparameterised queries, string-concatenated SQL, unsanitised input reaching dangerous sinks.
- Broken access control: missing auth/authorization checks on new endpoints or handlers.
- Cryptographic issues: weak hashing for secrets/passwords, plaintext secret storage, missing HTTPS-only flags on cookies.
- Hardcoded secrets/credentials in diffs.
- SSRF-shaped patterns: unsanitised user-supplied URLs in server-side requests.

GrepLoop is not OWASP-certified tooling. Findings are a first pass, not a security audit.

### 14.3 Performance
- Obvious N+1 query patterns in loops.
- Unbounded loops or recursive calls on user-controlled input.
- Large synchronous operations on a request/render path.
- (Web projects only, auto-detected) Core Web Vitals-relevant patterns: missing image dimensions, unoptimised image tags, render-blocking patterns.

### 14.4 Accessibility (web projects only, auto-detected)
- Missing alt text on images.
- Non-semantic interactive elements (`<div onClick>` instead of `<button>`).
- Missing form labels.
- Heading hierarchy skips.

This category is only activated when the diff touches frontend/markup files — skipped entirely for backend-only or CLI repos.

### 14.5 Style & maintainability
- Inconsistent naming conventions within changed files.
- Overly large functions/files relative to the rest of the codebase's own norms (thresholds inferred from the existing codebase, not hardcoded).
- Dead code / unused imports introduced by the diff.

## 15. LLM Backend Configuration

Both backend types are first-class, selectable at three levels (later overrides earlier):

1. **Global default** — set in server config.
2. **Per-organization** — admins set the org default.
3. **Per-repo** — overrides the org default for one repo.

| Backend | How it works | Tradeoffs |
|---|---|---|
| **Cloud API** | Claude / OpenAI-compatible endpoint. API key stored in org-level config (encrypted at rest in Postgres, never logged). | Higher review quality typically, requires network + API key + per-call cost, diff content leaves the customer's machine for the LLM provider. |
| **Local model** | Ollama (or any OpenAI-compatible local server) running on the same infrastructure. | Fully offline, no per-call cost, no data leaves the customer's network, quality depends on model size and hardware. |

GrepLoop doesn't bundle model weights — it talks to whichever local server the customer already has running, via a configurable endpoint URL. This keeps the GrepLoop image small and avoids it becoming a model-distribution problem.

For self-hosted customers in regulated industries, **local-backend-only mode** is a deployment option — cloud LLM calls are disabled at the config level, guaranteeing no code ever leaves the network.

## 18. Phase Sequencing

The indexing and agentic loop is the hard part. Build and validate it in stages:

**Phase A — User/org/repo model ships first.**
Multi-tenant auth wired (Better Auth + organization plugin). Repos can be registered, assigned to orgs, members added. No review yet — just the permissioned dashboard. Validates that the access model works before anything depends on it.

**Phase B — Get the index working (no agent yet).**
Tree-sitter parsing + call graph + Postgres storage (no LLM). Output: a "repo index status" view that shows indexed symbol count and edge count. Validate the graph is correct on a known small repo before touching the LLM pipeline.

**Phase C — Add docstring generation and embedding.**
Wire in the LLM call per symbol and pgvector storage. Output: a semantic search box in the UI that returns matching symbols with file/line. Validate retrieval quality manually on a real codebase.

**Phase D — Replace single-pass review with agentic loop.**
Agent loop + tool implementations. Start with the simplest tool set (`get_callers`, `get_symbol`, `search_codebase`) and add git history tools after the core loop works.

**Phase E — Evidence chains and confidence scoring.**
Evolve the report format from bare findings to the evidence chain format specified in Section 12. Reporting-layer change, not architecture.

**Phase F — Tune and calibrate.**
Run the full pipeline against real repos. Measure false positive rate and catch rate manually. Tune prompts, token budgets, and traversal depths based on real results.

## 19. Open Questions

- **Per-user vs per-org LLM API keys.** Org-level keys are simpler (one admin manages billing). Per-user keys let individuals bring their own. Decide after Phase D.
- **Embedding model for local backend.** `nomic-embed-text` is the current best local embedding model for code, but this is a fast-moving space. Re-evaluate at build time.
- **Max agent iterations.** Default of 10 is a guess. Needs empirical validation — too low and the agent stops before following an important lead; too high and runaway loops on deeply connected codebases cost too much on cloud backends.
- **PHP/SilverStripe-specific tree-sitter grammar.** tree-sitter-php exists and is mature. SilverStripe's `.ss` template files have no grammar — treat as plain text for MVP (they're presentation, rarely the source of the bugs this tool is built to catch).
- **Repo access auditing.** Should admins see a "who accessed this repo's reviews" log? Probably yes for compliance-sensitive customers, but defer to post-MVP.
- **Single-tenant vs multi-tenant install.** Even though the schema supports multiple orgs, MVP deployments will typically have one org per install (one customer = one install). Multi-org-per-install is supported by the schema but not a primary MVP use case.
